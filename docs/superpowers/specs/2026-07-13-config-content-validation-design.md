# 配置中心内容语法校验设计

## 背景

配置中心当前在创建、更新、回滚和导入配置时，会把配置值直接写入 etcd。若 YAML 等结构化配置未编辑完整，写入仍会成功，使用该配置的服务随后可能因解析失败而无法启动。

本次改动在配置写入前，根据配置 key 的文件后缀校验全部内容能否被对应解析器完整解析。

## 范围

首期支持以下后缀，匹配时不区分大小写：

- `.yaml`、`.yml`
- `.json`
- `.toml`

只校验语法，不校验业务必填字段或配置结构。无后缀及其他后缀仍视为普通文本，不做语法校验。

## 方案

在 service 层提供独立的配置内容校验器，并在所有配置中心写入路径调用。校验器只依赖 key 和 value，负责识别类型并返回可识别的校验错误。

选择 service 层而不是 handler 层，是为了让 HTTP 创建、HTTP 更新、版本回滚和批量导入遵循同一规则，避免未来出现绕过校验的写入入口。校验不会放到通用 etcd client 中，以免影响普通 KV、网关和服务注册等非配置中心数据。

## 校验行为

1. 使用 key 最后一级文件名的后缀判断配置类型，并将后缀转为小写。
2. YAML 使用流式解码器持续读取到 EOF，确保所有 YAML 文档及尾部内容都已被解析。
3. JSON 解码一个完整 JSON 值后，再次读取并确认到达 EOF，拒绝尾部残留内容或多个连续 JSON 值。
4. TOML 使用 TOML 解析器解析完整输入。
5. 未识别的后缀直接通过。

解析器原始错误会被包装为配置校验错误，保留行列等定位信息。JSON 标量、YAML 标量以及空 YAML/TOML 是否有效，以对应解析器的语法规则为准。

## 写入链路

- `Create`：校验通过后再查询并写入 etcd。
- `Update`：校验通过后再读取旧值并写入 etcd。
- `Rollback`：复用 `Update`，因此旧版本内容也必须通过当前校验规则。
- `Import`：解析导入文件后逐项按 key 后缀校验；非法项加入失败列表且不写入，其他合法项继续处理。dry-run 同样执行逐项校验并准确统计成功与失败数量。

任何校验失败都不得产生 etcd 写入或版本记录。

## 接口错误处理

创建、更新和回滚遇到配置校验错误时，handler 返回参数错误码，并将包含配置类型及解析器定位信息的消息返回前端。其他 service 错误继续使用现有错误映射。

批量导入维持现有部分成功语义，失败列表记录带原因的条目，便于用户定位具体 key 和语法错误。

## 测试

使用表驱动单元测试覆盖：

- 合法 YAML、YML、JSON、TOML。
- YAML 缩进、引号或流式括号不完整。
- JSON 括号、引号不完整以及合法值后的尾部垃圾。
- TOML 引号、数组或表声明不完整。
- 大小写混合后缀。
- 无后缀和未知后缀不做校验。
- 多文档 YAML 会完整读取，后续文档错误不能被忽略。
- 校验错误可被 handler 识别为参数错误。
- 导入 dry-run 和实际导入对非法配置的统计一致。

## 非目标

- 不校验配置必须包含哪些字段。
- 不推断无后缀配置的内容类型。
- 不修改前端编辑器的语言识别逻辑。
- 不校验配置中的 URL、端口、账号等业务值是否可用。
