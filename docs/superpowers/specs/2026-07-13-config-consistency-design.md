# 配置中心一致性加固设计

## 背景

配置中心当前通过“先查后写”修改 etcd，再单独写数据库版本记录。并发请求可能覆盖彼此；版本记录失败时 etcd 已生效但历史缺失；导入路径还会忽略 etcd 查询和版本写入错误。回滚按 revision ID 取值，但没有确认 revision 属于请求中的环境和 key。

## 范围

本设计覆盖：

- 创建、更新、删除配置的 etcd 原子条件写入。
- etcd 成功但版本记录失败时的条件补偿。
- 回滚版本归属校验。
- 导入路径复用一致的写入规则和错误统计。
- 类型化冲突与一致性错误。
- 可替换的 etcd 接口和自动化测试。

本次不引入 Outbox、后台协调器、数据库结构迁移或前端 `mod_revision` 字段。

## etcd 客户端边界

配置服务不再依赖具体的 `*etcd.Client`，而依赖只包含所需操作的接口。etcd 包提供以下原子能力：

- 仅在 key 不存在时创建。
- 仅在 key 的 mod revision 等于预期值时更新。
- 仅在 key 的 mod revision 等于预期值时删除。
- 仅在最近一次写入 revision 仍匹配时补偿恢复旧值或删除新 key。

接口返回写入前状态、写入后的 etcd revision 和是否因比较失败发生冲突。

## 创建、更新与删除

创建使用 etcd transaction 比较 `version(key) == 0`。比较失败返回 key 已存在，不产生版本记录。

更新和删除先读取当前 key 及 mod revision，再使用 transaction 比较 mod revision。读取后发生并发修改时比较失败，返回冲突错误，不覆盖新值。

更新不存在的 key 保持现有 upsert 语义，通过“key 不存在”条件创建并记录 update 动作；创建接口仍严格拒绝已存在 key。

## 版本记录与补偿

etcd 条件写入成功后创建数据库版本记录。数据库写入失败时，服务使用刚完成写入的 etcd revision 作为比较条件：

- 创建操作补偿为删除新 key。
- 更新和删除补偿为恢复写入前的值及原 lease（如有）。

只有 key 仍处于本次写入产生的 revision 时才补偿，避免覆盖其他并发请求。补偿成功时返回版本记录错误，并保证最终配置值恢复；补偿比较失败或 etcd 错误时返回类型化的一致性错误，明确指出需要人工检查或同步修复。

跨 etcd 与数据库无法获得真正的单事务原子性，因此补偿失败必须记录结构化错误，不能伪装为普通失败。

## 回滚

回滚获取 revision 后，必须同时满足：

- revision 的 `environment_id` 等于目标环境 ID。
- revision 的 key 等于请求 key。
- revision 的 action 不是 delete；删除记录不允许通过当前“恢复某版本”接口回滚，避免把空 value 误写为有效配置。

不匹配时返回 revision 不存在，避免泄露其他环境或 key 的历史信息。匹配后复用更新流程和配置语法校验。

## 导入

导入先解析并校验所有配置项，然后逐项调用与创建/更新相同的内部 upsert 流程：

- etcd 查询错误记为失败，不继续写该项。
- 条件写冲突记为失败。
- 版本记录失败和补偿结果写入具体失败原因。
- 只有 etcd 写入与版本记录都成功才增加 Success。

dry-run 只执行格式和配置语法校验，不访问 etcd 或数据库。

## 错误模型

新增哨兵或类型化错误：

- key 已存在。
- 并发写入冲突。
- revision 不存在或不属于目标资源。
- 配置语法错误。
- 版本持久化失败且补偿成功。
- 补偿失败导致状态可能不一致。

handler 使用 `errors.Is/As` 映射现有响应码，并为并发冲突增加明确业务错误码。禁止使用 `err.Error() == ...` 分支。

## 测试

- 两个并发创建只能有一个成功。
- 更新和删除在 mod revision 变化时返回冲突。
- 版本记录失败后创建被删除、更新/删除恢复旧值。
- 补偿期间再次发生并发写入时不覆盖新值，并返回一致性错误。
- 回滚拒绝其他环境和其他 key 的 revision。
- 合法回滚复用语法校验与版本记录。
- 导入不会把查询失败或版本失败计为成功。
- dry-run 不调用 etcd 或 repository。

## 非目标

- 不提供跨 etcd 和数据库的强分布式事务。
- 不新增后台 Outbox 或自动重试任务。
- 不在本轮实现前端长编辑会话的显式版本号冲突提示。
