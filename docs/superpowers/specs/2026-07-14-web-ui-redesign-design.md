# ETCD Manager Web UI 全站改版设计

**日期：** 2026-07-14

**状态：** 已确认

**目标：** 在不改变现有业务能力、API 契约、路由和权限行为的前提下，将整个 Web 管理端重构为清爽、专业、高密度的云控制台界面。

## 1. 背景与问题

当前前端基于 React 18、TypeScript、Vite、Ant Design 5 和 Zustand，功能完整，但视觉主要依赖 Ant Design 默认样式，并在页面和主布局中大量使用内联样式。由此产生以下问题：

- 缺少统一的品牌语言，登录页、导航、顶栏和业务页面像独立拼装的组件。
- 页面标题、工具栏、卡片、表格和操作按钮缺少稳定层级，跨模块使用成本高。
- 内容区被单个白色大容器包裹，监控指标、筛选区和数据区之间的层次不清晰。
- 状态色、间距、圆角、阴影、字体和深色主题缺少统一 token。
- 加载、空数据、局部错误和危险操作反馈不够系统化。

本次改版覆盖登录页、全局布局和全部业务页面，不增加新业务功能，不改后端。

## 2. 设计目标

### 2.1 核心目标

1. 建立一致的「清爽云控制台」视觉语言。
2. 面向大屏桌面运维场景，提高信息扫描与操作效率。
3. 让所有业务模块遵循一致的页面骨架与交互语义。
4. 保留浅色、深色和跟随系统三种主题模式。
5. 保持现有 API、状态管理、RBAC、环境隔离和路由行为不变。

### 2.2 非目标

- 不新增或修改后端 API。
- 不改变业务流程、权限模型和数据结构。
- 不为手机和平板提供完整响应式布局。
- 不引入新的大型 UI 组件库，也不替换 Ant Design。
- 不进行与视觉改版无关的前端架构重写。

## 3. 目标平台与尺寸

- 主要目标：1440px 及以上的大屏桌面浏览器。
- 可用下限：1280px；低于该宽度不保证完整体验，但不得出现页面崩溃。
- 1920px 宽屏下内容应充分利用空间，表格和监控视图不应被不必要地限宽。
- 页面采用高信息密度：常规表格行高约 44px，工具栏单行优先。

## 4. 视觉方向

采用已确认的 A 方案「清爽云控制台」。

### 4.1 视觉语言

- 深海军蓝侧栏配明亮的浅灰蓝内容背景。
- 蓝色用于主操作、选中状态和可交互强调。
- 青绿色用于健康、在线和成功状态。
- 橙色用于警告，红色用于错误与破坏性操作。
- 内容卡片采用轻边框、弱阴影和 10–14px 圆角，避免过度悬浮。
- 标题使用清晰的无衬线字体；Key、ID、Endpoint、版本号和配置内容使用等宽字体。
- 浅色主题为默认呈现；深色主题使用独立语义 token，不做简单颜色反转。

### 4.2 设计 token

通过 Ant Design `ConfigProvider` 与全局 CSS 变量集中定义：

- 品牌色与交互色。
- 页面、容器、悬浮层和侧栏表面色。
- 成功、警告、错误、信息等语义色。
- 字号、行高、字重和等宽字体栈。
- 4px 基础间距体系。
- 圆角、边框、阴影和过渡时长。

业务页面不得新增任意硬编码主题色；确有特例时必须通过语义变量表达。

## 5. 全局框架

### 5.1 侧栏

- 固定宽度约 216–224px，使用深色背景。
- 顶部展示 `etcd manager` 品牌标识和简短副标题。
- 菜单分为三组：
  - 资源管理：集群概览、KV 管理、配置中心。
  - 服务治理：网关服务、gRPC 服务。
  - 系统管理：用户管理、角色管理、审计日志。
- 菜单可见性继续由当前 RBAC 逻辑决定；无权限项不渲染。
- 当前项使用明显但克制的蓝色选中态。
- 底部可展示集群健康摘要；数据不可用时隐藏该摘要，不阻塞导航。

### 5.2 顶栏

- 高度约 60–64px，固定在工作区顶部。
- 左侧显示当前位置或页面面包屑。
- 右侧依次放置环境选择、主题切换、辅助入口和用户菜单。
- 环境切换仍使用现有 environment store；环境管理入口由现有写权限控制。
- 用户菜单保留修改密码和退出登录。

### 5.3 内容区

- 内容区使用浅灰蓝页面背景，不再将所有内容包在单一白色大容器中。
- 标准页面顺序：
  1. PageHeader：模块名、标题、说明和主操作。
  2. 可选摘要指标或重要告警。
  3. 筛选与工具栏。
  4. 数据卡片、表格、树或编辑器。
  5. 分页、批量操作或补充说明。
- 页面主操作固定在 PageHeader 右侧；同一页面只允许一个视觉主按钮。

## 6. 公共组件边界

在现有 Ant Design 组件之上建立轻量公共层，避免业务页重复实现视觉结构。

### 6.1 必要公共组件

- `AppBrand`：品牌图形与产品名称。
- `PageHeader`：页面眉题、标题、说明和主操作。
- `PageToolbar`：搜索、筛选、刷新、视图切换和批量操作布局。
- `MetricCard`：监控与摘要指标卡。
- `StatusBadge`：健康、在线、警告、异常、禁用等统一状态表达。
- `EmptyState`：空数据说明与可选主操作。
- `ErrorState`：局部错误说明与重试动作。
- `CopyableCode`：Key、ID、Endpoint 等等宽内容与复制能力。
- `SectionCard`：带标题、补充说明和可选动作的内容容器。

### 6.2 组件原则

- 组件只负责表现和局部交互，不直接调用业务 API。
- 数据加载、权限判断和业务状态继续由现有页面与 store 管理。
- 公共组件接口使用明确的语义属性，禁止依赖页面内部实现。
- 不抽象只使用一次且结构简单的页面片段。

## 7. 业务页面模式

### 7.1 资源浏览型

适用于 KV 管理和配置中心。

- 搜索、刷新、新建和列表/树形切换放在统一工具栏。
- 树形结构负责定位资源，内容区负责查看或编辑。
- 配置、Key 和 Value 使用等宽字体。
- 新建与编辑继续使用 Monaco Editor；弹窗尺寸和操作区统一。
- 版本历史、导入、导出和回滚保留原有行为，危险动作明确显示影响范围。

### 7.2 状态监控型

适用于集群、网关和 gRPC 页面。

- 页面顶部使用摘要指标展示关键健康信息。
- 正常状态克制呈现，异常状态获得更高视觉优先级。
- 服务与实例信息在表格中使用主次两层文字，减少横向列数量。
- Leader、Learner、健康、异常和禁用等状态使用统一徽标。
- 刷新操作位于 PageHeader 或工具栏固定位置。

### 7.3 管理与审计型

适用于用户、角色和审计页面。

- 筛选项紧凑排列，表格作为视觉主体。
- 行内操作位于末列；破坏性操作使用红色文本或按钮并二次确认。
- 权限矩阵保持现有逻辑，通过分组、说明文字和层级改善可读性。
- 审计事件突出资源、操作者、时间和结果，技术标识使用等宽字体。

## 8. 登录页

- 使用左右分栏大屏布局。
- 左侧为深色品牌区，展示产品名称、简短价值说明和能力标签。
- 右侧为明亮登录表单，包含用户名、密码、提交按钮和安全提示。
- 登录逻辑、错误处理和登录后跳转保持不变。
- 深色主题下表单区使用对应深色表面 token，保证输入框与提示文本对比度。
- 不增加注册、找回密码或第三方登录能力。

## 9. 数据流与权限

改版不改变现有数据流：

1. 页面通过现有 API 模块请求数据。
2. 认证、主题和当前环境继续由 Zustand store 管理。
3. 页面将业务状态传给公共表现组件。
4. 用户操作调用现有页面 handler 与 API。
5. API 成功后刷新局部数据或关闭弹窗；失败时保留已有内容并展示局部反馈。

`canRead`、`canWrite`、`isSuper`、`getVisibleMenuKeys` 和环境访问控制继续作为唯一权限来源。视觉层不得自行推断或复制权限规则。

## 10. 系统状态与错误处理

### 10.1 加载

- 首次加载使用与最终布局一致的骨架屏或局部 Spin。
- 已有数据刷新时保留内容，仅在刷新入口和相关区域显示加载状态。
- 避免因局部请求刷新导致整个页面闪烁或消失。

### 10.2 空数据

- 说明当前为何无数据，并提供与当前页面最相关的下一步。
- 只有具备写权限时才显示新建动作。
- 搜索无结果与系统尚无数据使用不同文案。

### 10.3 错误

- 首次加载失败且无缓存内容时展示页面级错误和重试入口。
- 刷新失败且已有内容时保留内容，使用局部 Alert 或 message 提示。
- 后端错误消息可安全展示时沿用现有文本，否则使用模块级默认错误文案。

### 10.4 危险操作

- 删除、回滚和配置恢复必须二次确认。
- 确认内容明确资源名称、作用环境和不可逆影响。
- 执行中锁定重复提交入口，完成后提供明确成功或失败反馈。

## 11. 页面范围

以下页面全部纳入改版：

1. 登录页。
2. 主布局与环境管理、密码修改、配置恢复弹窗。
3. 集群概览。
4. KV 管理及树形视图。
5. 配置中心及版本、导入导出、回滚流程。
6. 网关服务。
7. gRPC 服务。
8. 用户管理。
9. 角色管理。
10. 审计日志。

## 12. 实施顺序

1. 建立设计 token、基础 CSS 与 Ant Design 主题配置。
2. 实现公共表现组件。
3. 改造登录页和 MainLayout。
4. 改造集群概览并验证指标卡、状态徽标和表格模式。
5. 改造 KV 与配置中心，验证资源浏览型页面模式。
6. 改造网关和 gRPC，验证状态监控型页面模式。
7. 改造用户、角色和审计，验证管理型页面模式。
8. 统一弹窗、空状态、错误状态与深色主题细节。
9. 完成构建、测试与大屏视觉验收。

## 13. 验收标准

### 13.1 功能与构建

- 所有现有路由、API 调用、权限限制、环境选择和主题模式保持可用。
- TypeScript 检查和 Vite 生产构建通过。
- 不新增后端依赖，不更改 API 契约。

### 13.2 视觉与交互

- 1440px 和 1920px 下浅色、深色主题均无遮挡、溢出或不可读表格。
- 登录页、主布局和 8 个业务模块使用同一视觉语言。
- 页面标题、主操作、工具栏、表格、状态色和弹窗结构一致。
- 主要页面覆盖首次加载、已有数据刷新、空数据和错误状态。
- 键盘焦点、文本对比度和禁用态清晰可辨。

### 13.3 回归验证

- 登录、退出和修改密码。
- 环境切换与环境管理权限。
- 各模块读写权限及无权限菜单隐藏。
- KV 与配置 CRUD、配置版本与回滚。
- 网关和 gRPC 状态操作。
- 用户、角色与审计筛选操作。
- 浅色、深色和跟随系统主题切换。

## 14. 风险与控制

- **风险：** 一次性修改所有页面可能导致视觉与功能回归范围过大。

  **控制：** 按页面模式分批迁移，每一批完成构建与核心交互检查。

- **风险：** 大量现有内联样式与全局主题冲突。

  **控制：** 优先将公共布局和语义色迁移到 token 与样式类，避免高优先级覆盖。

- **风险：** 深色主题仅在最后处理会产生大量返工。

  **控制：** 每个公共组件实现时同时验证浅色与深色 token。

- **风险：** 视觉抽象过度导致页面实现复杂。

  **控制：** 只提取跨两个及以上页面复用、且语义稳定的组件。
