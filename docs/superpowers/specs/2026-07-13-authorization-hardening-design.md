# 授权与认证加固设计

## 背景

项目当前将角色允许访问的环境 ID 放入 Gin context，但除环境列表外，配置中心、网关和 gRPC 接口没有消费该授权范围。JWT 同时携带并直接信任登录时的超管和角色状态，用户删除、角色变更或超管转移无法在旧 Token 上立即生效。超管转移由两次独立数据库更新组成，中途失败可能留下不完整状态。

## 范围

本设计覆盖：

- 配置中心全部接口的环境级授权。
- 网关和 gRPC 服务的环境级前缀约束。
- JWT 验签后的当前用户状态加载。
- 超管转移的数据库事务。
- 对应的后端和前端 API 调整与自动化测试。

普通 KV 和 Watch 保持拥有 `kv` 模块权限即可访问全部 etcd key 的现有语义，不应用环境过滤。

默认 `admin/admin123` 和 `change-me-in-production` 保持不变。本次不实现服务端会话表、Token 吊销列表或密码修改后的 Token 失效。

## JWT 与当前用户

JWT Claims 保留用户 ID、签发时间和过期时间。验签时显式限定 HS256。签名和有效期通过后，中间件根据用户 ID 查询当前用户；用户不存在时拒绝请求，并把数据库中的最新 `is_super`、`role_id` 和用户名写入 Gin context。

权限中间件继续按最新角色 ID 查询权限与环境范围。因此删除用户、调整角色和转移超管会在下一次请求立即生效。

## 环境授权

`FilterEnvironments` 负责获取当前角色的允许环境 ID，并在 context 中保存明确的授权范围。新增集中式 helper：

- 超级管理员直接通过。
- 普通用户只有目标环境 ID 出现在允许列表中才通过。
- 缺少授权范围或环境不匹配时返回禁止访问错误。

配置中心 handler 在解析 `env` 后，通过环境服务解析环境实体，再调用 helper。以下入口必须覆盖：列表、创建、更新、删除、版本历史、回滚、导出和导入。

环境列表继续按允许环境 ID 过滤。

## 网关和 gRPC 边界

网关和 gRPC 查询接口不再信任客户端提供的任意前缀，改为接收环境名 `env`。handler 解析并授权环境后，使用环境实体中的 `key_prefix + gateway_prefix` 或 `key_prefix + grpc_prefix` 调用 service。

状态更新请求同时包含 `env` 和实例 key。handler 在授权环境后，校验 key 以该环境的对应服务前缀开头；不满足时拒绝，不访问 etcd。

前端 API 和页面改为传当前环境名，不再自行拼装可变前缀。

## 普通 KV 与 Watch

从普通 KV 和 Watch 路由移除 `FilterEnvironments`。两者继续使用 `RequirePermission("kv")`，其权限语义是可访问全部 etcd key。

## 超管转移事务

`UserService` 注入现有 `TransactionManager`。转移开始前验证降级角色和目标用户；随后在同一事务 context 中重新读取当前用户与目标用户，并依次完成旧超管降级、新超管升级。任意查询或更新失败时事务回滚。

JWT 中间件每次读取当前用户，因此事务提交后旧超管的下一次请求立即失去超管权限。

## 错误处理

新增类型化的未认证、禁止访问和资源不存在错误，handler 使用 `errors.Is/As` 映射现有业务错误码，不再通过错误字符串判断授权结果。

对外错误不暴露数据库内部细节。授权失败统一返回禁止访问，用户不存在或 Token 无效返回未认证。

## 测试

- JWT 只接受 HS256，并在每次请求读取当前用户。
- 已删除用户使用旧 Token 被拒绝。
- 超管转移后旧 Token 立即失去超管权限，新超管获得权限。
- 超管转移第二次更新失败时第一次更新回滚。
- 普通用户只能访问授权环境的全部配置接口。
- 网关/gRPC 查询只能使用授权环境生成的前缀。
- 网关/gRPC 状态更新拒绝环境前缀之外的 key。
- 普通 KV 和 Watch 不应用环境范围，但仍要求 `kv` 模块权限。

## 非目标

- 不改变默认管理员和默认 JWT secret。
- 不引入服务端 Session、Refresh Token 或 Token 黑名单。
- 不把普通 KV key 强行归属到环境。
